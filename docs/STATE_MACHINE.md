## 1. 订单状态枚举（建议内部值）

以下为**推荐内部标识**，实现可用 `VARCHAR` 或 `TINYINT` 映射，须与 `docs/SCHEMA.md` `orders.status` 一致。

| 内部值（示例） | 业务含义 | 用户可见（参考 P0-1） |
|----------------|----------|------------------------|
| `PENDING_GRAB` | 待抢单，在大厅可见 | 等待接单 / 待陪玩接单 |
| `ACCEPTED` | 已被某陪玩认领 | 已接单 |
| `IN_SERVICE` | 服务进行中 | 服务中 |
| `COMPLETED` | 正常结束 | 已完成 |
| `CANCELLED` | 取消或强制关闭 | 已取消 |

**终态**：`COMPLETED`、`CANCELLED`。**不再转出**（除非负责人书面批准管理端「复活」类操作，一期默认不做）。

## 2. 合法迁移（v0 草案）

| 自 | 至 | 触发方 | 说明 |
|----|-----|--------|------|
| （创建） | `PENDING_GRAB` | 玩家 / 系统 | 下单成功 |
| `PENDING_GRAB` | `ACCEPTED` | 陪玩（抢单） | P0-2 行为 C；须并发互斥 |
| `PENDING_GRAB` | `CANCELLED` | 玩家 | 仅**无人认领**时允许（P0-1 §6） |
| `PENDING_GRAB` | `CANCELLED` | 管理员 | 兜底取消 |
| `ACCEPTED` | `IN_SERVICE` | 陪玩 | 「开始服务」类操作 |
| `IN_SERVICE` | `COMPLETED` | 陪玩 或 管理员确认 | 以 P2-1 为准 |
| `ACCEPTED` | `CANCELLED` | 管理员 | 玩家不可自助取消已接单及之后（P0-1 §6） |
| `IN_SERVICE` | `CANCELLED` | 管理员 | 异常收尾 |
| `ACCEPTED` | `CANCELLED` | 玩家 | **默认不允许**（与 P0-1 一致）；若 P2-1 放开须单独列 |

**一期默认不提供**：`PENDING_GRAB` → 由管理员**指派**到某陪玩（P0-3 §4.1 选 A）。若未来开放行为 D，须与 P0-2 §1.1 **串行裁决** 同表补充一行。

**不做**：`ACCEPTED` / `IN_SERVICE` → 「改派」到另一陪玩（P0-3 §4.3）。

## 3. 触发动作与责任

| 动作 | 用户端 | 陪玩端 | 管理端 | 系统 |
|------|--------|--------|--------|------|
| 创建订单 | ✓ | — | — | 可记创建日志 |
| 抢单成功 | — | ✓ | — | 并发裁决 |
| 开始服务 | — | ✓ | 可选 | — |
| 完成订单 | — | ✓ | 可选 | — |
| 玩家取消（仅待抢单） | ✓ | — | — | — |
| 取消 / 强制状态修正 | — | — | ✓ | — |

所有状态变更须在 **`order_logs`**（或等价）留痕，Phase 2 验收（总计划 Phase 2）。

---

## 4. 服务端校验要点（摘要）

1. **任意迁移**：当前状态必须在边允许表中；否则返回 **409** 类业务码（见 `docs/API_CONTRACT.md` §4）。
2. **抢单**：同一 `orderId` 仅允许一人成功；失败方明确错误（P0-2 §1.1）。
3. **权限**：玩家仅能操作本人订单；陪玩仅能操作已归属自己的单；管理端鉴权独立。
4. **玩家取消**：仅 `PENDING_GRAB` 且 `companion_user_id IS NULL`（或等价）时允许。
