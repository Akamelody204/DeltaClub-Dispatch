# 三角洲俱乐部陪玩派单小程序 · 中小团队快速落实计划

> 定位：**vibe coding** —— 少开会、多可运行产物；先闭环 MVP，再迭代体验与商业化。  
> 适用：3～5 人小团队，4～10 周内跑出可内测版本（视是否接支付而定）。

**用户端范围（当前确认）**：**仅微信小程序**（不排期支付宝小程序、H5 用户端、App）。**PC 管理后台**（浏览器）照常，不计入「用户端多端」。细节与选型理由见 **`技术栈选型记录.md`** 文首「项目范围」。

**领头人执行顺序**（先谁后谁、阶段门禁、与各负责人任务编号对齐）：见 **`领头人-开发步骤指导.md`**。

**当前阶段（仓库共识）**：**Phase 1 · 骨架**（Phase 0 已于 2026-04-04 收口，见 **`Phase0-收口与Phase1开启.md`**）。

---

## 1. 原则（团队共识，写进群公告即可）

1. **MVP 先跑通**：能「下单 → 派单/抢单 → 状态变更 → 后台看得见」比页面精致更重要。  
2. **单仓优先**：小程序 + 后端 + 管理端可 monorepo 或两仓（`miniapp` + `server`），避免过早微服务。  
3. **接口契约口头 + 文档补**：用一份飞书/Notion 维护「接口表 + 状态机」，改了就改一行，不追求完整 OpenAPI 再开发。  
4. **主分支可部署**：每天至少有一个环境能打开小程序连上真后端（哪怕数据是 mock 混用）。  
5. **微信规范红线**：类目、文案、外链、私下交易引导 —— 产品每次发版前过一遍审核要点。

## 2. 建议角色分工（可一人多职）

| 角色 | 职责 |
|------|------|
| 全栈 / 后端 | 用户鉴权、订单与派单、状态机、管理端接口、部署 |
| 小程序 | 玩家端 + 陪玩端页面（或先做同端角色切换） |
| 产品兼运营 | 流程、文案、验收标准、与俱乐部对齐规则 |
| 设计（可兼职） | 仅主流程 4～6 个关键页高保真，其余用组件库 |

下文 **「详细开发流程计划」** 中，为省事可用代号：**后端**、**小程序**、**产品**、**设计**（与上表一一对应；一人多职则合并执行）。

## 3. 详细开发流程计划（分阶段 · 谁做什么）

> **说明**：本段先写 **Phase 0～3** 的可执行任务级拆解（可直接拆 issue）；**Phase 4** 及 **Phase 5+（全量延伸）** 先给提纲，后续随立项补充日期、工时与负责人姓名即可。  
> **依赖**：产品先定「派单模式（抢单 / 指派）」与「一期不做清单」，否则后端与小程序并行时容易返工。

### Phase 0：对齐与准备（约 3～5 天）

| 谁 | 要做什么 | 产出 / 完成标志 |
|----|----------|------------------|
| **产品** | 与俱乐部对齐：用户路径、派单规则、定价展示方式、是否一期做「争议/退款」 | 流程说明 1 页 + **一期范围**书面确认（含不做项） |
| **产品** | 查微信小程序类目、主体资质、敏感词与导流红线；列审核自检表 | 类目与文案注意事项清单 |
| **后端** | 定仓库结构、环境（dev/staging）、域名与 HTTPS、数据库与 Redis 规划；搭空项目可运行（健康检查接口） | 空仓库 + 本地/服务器可访问 API + README 部署草稿 |
| **小程序** | 注册/绑定小程序 AppID、开发者工具打通；搭空项目 tabBar/页面路由骨架 | 可编译运行的空壳工程 |
| **设计** | 与产品对主路径：玩家下单、陪玩接单、订单详情、我的（可选管理端信息架构） | 线框或关键页结构共识（不必完稿） |
| **全员** | 对齐技术选型（与本文第 4 节一致或记录偏差）；约定接口文档位置（飞书/Notion） | 选型结论 1 段话 + 文档链接 |

**Phase 0 退出标准**：范围冻结（可小改但不大改）；三端空工程都能跑；产品交付审核自检表。

---

### Phase 1：骨架（约 1～2 周）

| 谁 | 要做什么 | 产出 / 完成标志 |
|----|----------|------------------|
| **后端** | 微信登录：`code` → `session`；用户表（openid、角色、基础资料）；简单鉴权中间件 | 登录接口 + 测试通过（真机或开发者工具） |
| **后端** | 订单表初版（可先少字段）、订单 CRUD 草稿接口；错误码与统一返回格式 | 接口可被 Postman/前端调用 |
| **后端** | （若管理端同栈）预留管理员鉴权方式（账号密码或同微信角色） | 管理端能调通的占位接口 |
| **小程序** | 登录页/启动页：拉 code、换 token、存 storage；无 token 拦截跳转 | 真机可登录拿到身份 |
| **小程序** | 玩家端骨架页：首页、下单（可先静态表单）、订单列表、我的；陪玩入口或 Tab 分流（按产品定） | 页面可切换，请求带 token |
| **小程序** | 陪玩端骨架：待接单列表（可先 mock）、进行中订单入口 | 路由与角色可见性符合产品规则 |
| **产品** | 写 **接口表 v0**（路径、字段、谁调）；与后端对一遍字段名 | 接口表可开发 |
| **产品** | 文案：按钮、空状态、错误提示初稿；订单状态用户可见名称 | 文案贴进文档或 issue |
| **设计** | 4～6 个关键页视觉（或仅组件规范：主色、按钮、列表卡片） | 小程序可按稿或规范开工 |
| **后端** | staging 部署一版；小程序后台配置 **request 合法域名** | 小程序连 staging 成功 |

**Phase 1 退出标准**：真实登录链路打通；玩家/陪玩能看到对应空壳页；后端 staging 可连。

---

### Phase 2：派单闭环 MVP（约 2～3 周）

| 谁 | 要做什么 | 产出 / 完成标志 |
|----|----------|------------------|
| **产品** | 冻结 **订单状态机**（与开发一致）；异常流：谁可取消、何时可取消 | 状态机图 + 文档 |
| **后端** | 实现状态机迁移（仅允许合法 transition）；所有改状态走同一套服务方法 | 单元测试或集成测试覆盖主路径 |
| **后端** | **派单策略（二选一先落地）**：抢单大厅 *或* 管理员指派；并发下 **防重复接单**（Redis 锁或乐观锁） | 压测或并发脚本下不出现双接单 |
| **后端** | 接口：创建订单、列表、详情、陪玩抢单/接单、推进状态；管理端查询与改派/取消 | 与接口表一致 |
| **后端** | 操作日志：关键状态变更记一条（谁、何时、原状态、新状态） | 后台可查或 DB 可查 |
| **小程序** | 玩家：下单表单校验、提交订单、订单列表与详情、取消（若一期要做） | 端到端玩家侧可用 |
| **小程序** | 陪玩：大厅列表/抢单 *或* 仅「我的待接」；接单按钮与状态刷新 | 端到端陪玩侧可用 |
| **小程序** | 统一 loading / 错误 toast；防止重复点击提交 | 主流程无明显 UX 事故 |
| **后端** | 管理后台 API：订单筛选、详情、指派陪玩、强制状态修正（权限校验） | 与前端联调通过 |
| **小程序**或**另开管理端人力** | 管理后台 Web：登录、订单表、详情页、指派/改状态（若团队无专人，可由后端兼 Vue 或延后极简版） | 俱乐部能操作最小闭环 |
| **产品** | 每 2～3 天验收一次主路径；记录 bug 优先级 | 已知问题列表 |
| **设计** | 补边角：空列表、失败态、重要按钮状态 | UI 与实现一致 |

**Phase 2 退出标准**：玩家下单 → 陪玩接单（或管理指派）→ 服务中 → 完成/取消 全链路真实数据跑通；后台可见订单与日志。

---

### Phase 3：可用性与上线准备（约 1～2 周）

| 谁 | 要做什么 | 产出 / 完成标志 |
|----|----------|------------------|
| **后端** | 列表分页、筛选参数约定；敏感接口限流（按 IP/用户可选） | 大数据量列表不爆 |
| **小程序** | 订阅消息：申请模板、对接发送节点（接单、重要状态变更）；用户拒绝授权时的降级提示 | 关键节点能触达（在微信规则内） |
| **小程序** | 列表分页、下拉刷新；空状态与网络错误文案按产品稿 | 体验达标 |
| **产品** | 走完整 **审核自检表**；准备小程序提审说明、截图、测试账号 | 可提交审核的材料包 |
| **后端** | 生产环境配置、备份策略（数据库定时备份）、密钥不进仓库 | 上线 checklist 勾完 |
| **全员** | 选一天 **预演验收**（俱乐部的人按脚本点一遍） | 预演问题清单 → 修完再发版 |

**Phase 3 退出标准**：通过内部/俱乐部预演；可提交微信审核（若一期要上线）；staging 与生产配置分离清晰。

## 4. 技术选型（轻量、好招人、易部署）

| 层 | 推荐 | 备选 |
|----|------|------|
| 小程序 | **原生** 或 **uni-app (Vue3)** | Taro |
| 管理后台 | **Vue3 + Element Plus** | React + Ant Design |
| 后端 | **Node (NestJS)** 或 **Java (Spring Boot)** | Go (Gin) |
| 数据库 | **MySQL 8** | PostgreSQL |
| 缓存 / 锁 | **Redis** | 派单防重复、简单队列、session |
| 文件 | **腾讯云 COS**（或阿里云 OSS） | 头像、凭证图 |
| 实时（可选二期） | **WebSocket** 同域或独立网关 | 一期可用轮询 + 订阅消息 |

**支付**：一期可「线下结算 + 系统记账」；确定商业化再接 **微信支付** 与分账方案。

> **已定稿的倾向选型、对他人推荐的批判与分期采纳说明**：见同目录 **`技术栈选型记录.md`**。

## 5. 分期交付概要（按周推进，可压缩/拉长）

> **与第 3 节关系**：本节是「阶段一览」；**谁做什么、产出与退出标准** 以 **§3 详细开发流程计划** 为准。

### Phase 0：对齐（3～5 天）

- 业务流程图：玩家下单、陪玩接单/被派单、取消、完成、争议（是否一期做）。  
- 名词统一：订单状态、角色名、派单规则（抢单 / 指派 / 混合）。  
- 小程序类目与主体资质确认（避免做完不能上线）。

### Phase 1：骨架（约 1～2 周）

- 微信登录、角色（玩家 / 陪玩 / 管理员）模型。  
- 空壳页面：首页、下单、订单列表、我的；陪玩端「待接/进行中」。  
- 后端：用户表、订单表、基础 CRUD、测试环境 + 域名 HTTPS。

### Phase 2：派单闭环 MVP（约 2～3 周）

- 订单状态机：`待派单 → 已接单 → 服务中 → 已完成 / 已取消`。  
- 一种派单策略先写死（建议先 **抢单大厅** 或先 **管理员指派**，二选一）。  
- **Redis 分布式锁** 或 DB 乐观锁：防止一单多接。  
- 管理后台：订单列表、改状态、陪玩账号开关。

### Phase 3：可用性（约 1～2 周）

- **订阅消息** 关键节点（接单、状态变更）。  
- 列表分页、异常提示、空状态。  
- 简单评价（可选）。

### Phase 4：商业化与增强（按需）

- 微信支付、结算对账、分账。  
- 风控：黑名单、异常订单标记。  
- 实时通知升级 WebSocket。

## 6. 核心数据与接口（开发 checklist）

**必须有**

- 用户（openid、角色、陪玩资料审核状态）  
- 服务 SKU（游戏区服、时长、价格展示 —— 一期可简化为固定几项）  
- 订单（状态、玩家、陪玩、创建/接单时间、备注）  
- 操作日志（后台审计，至少关键状态变更）

**接口最小集**

- 登录 / 刷新会话  
- 创建订单、订单列表、订单详情  
- 陪玩：可抢列表 / 接单 / 更新状态  
- 管理：订单查询、指派、强制完结/取消  

---

## 7. Vibe 工作流（少形式，多同步）

- **每日**：15 分钟站会（语音即可）—— 昨天合并了什么、今天卡什么、接口是否 breaking。  
- **分支**：`main` 保护 + `feat/*`；小功能直接 PR 自己 merge（团队小时）。  
- **环境**：`dev`（随便造）+ `staging`（给俱乐部演示）；配置用环境变量，不写死密钥。  
- **文档**：只维护一份「状态机 + 接口表 + 部署说明」，其它写在代码 README。

---

## 8. 风险清单（提前写在计划里，避免背锅）

| 风险 | 应对 |
|------|------|
| 微信审核不通过 | 提前查类目与案例；敏感词与导流话术规避 |
| 派单并发重复接单 | 锁 + 状态机单线转移 + 单元测试覆盖 |
| 需求蔓延 | 「一期不做」清单书面确认（语音/分账/多俱乐部等） |
| 一人请假阻塞 | 核心模块两人可交叉提交；接口表共享 |

---

## 9. 验收标准（MVP 算「落实」的底线）

1. 真实微信环境可登录并区分玩家/陪玩（或测试号流程）。  
2. 玩家能下单，陪玩能接到单，订单状态正确推进到完成。  
3. 管理后台能查到订单并做最低限度干预（取消/改派）。  
4. 部署文档齐全：另一台机器按步骤能拉起后端与后台。
