# 设计负责人工作计划（对总计划负责）

> **角色**：设计（可兼职，见总计划 §2）。  
> **责任边界**：主流程关键页视觉与组件规范未按期交付、与产品主路径不一致、或边角态（空/错/禁用）未覆盖导致小程序反复改稿或体验事故，由设计侧牵头补交付与对齐；**切图标注、实现细节**与研发共建，但**视觉真源、规范版本**以本文约定产出为准。  
> **依据文档**：`../三角洲陪玩派单小程序-快速落实计划.md`、`../技术栈选型记录.md`。  
> **执行交付索引**：`dev/README.md`（与 `D0-x` / `D1-x` 任务一一对应的落地说明）。  
> **跨端对接（设计侧仅维护本目录）**：`与其他端的对接说明-非design目录曾由会话改动清单.md`（供产品/小程序/后端核对非 `plan/design` 曾改动与应读文档）。

---

## 1. 总原则（我承诺遵守）

1. **MVP 优先于精致**：总计划 §1 要求先跑通闭环；设计聚焦 **4～6 个关键页高保真**，其余页面以 **组件库 + 规范** 支撑，不追求全站逐页精修。  
2. **单一真源**：色板、字号阶梯、圆角、按钮层级、列表卡片结构定稿后写入 **设计规范一页**（或设计稿内「组件」页），变更记版本号并 @ 小程序负责人。  
3. **微信与审核语境**：不设计诱导私下交易、过度营销的视觉话术；关键文案以 **产品文档** 为准，设计稿中占位符需与产品最终稿同步替换。  
4. **双端范围**：**用户端仅微信小程序**；**管理后台 Web** 的 IA 已单独落在 **`dev/D0-3-管理端信息架构.md`**（可选签收）；若本期不做管理端演示，可勾选「本期跳过」不阻塞小程序主路径。

### Phase 4 及以后（提纲）

- **支付结果页、收银台**（若立项）：独立补关键页与错误重试态。

- 不在本期承诺全站重设计；需求蔓延时坚持总计划 **「一期不做清单」**，新增页面走变更单。

## 4. 风险与应对（设计侧）

| 风险 | 应对 |
|------|------|
| 兼职时间不足导致 Phase 1 阻塞 | Phase 0 结束即锁定 4～6 页清单；优先 **规范 + 2 页** 最小包，余页迭代补充 |
| 产品改主路径导致返工 | 结构变更必须书面/文档更新版本；大改走变更记录，不静默改稿 |
| 组件库与设计稿差异大 | 规范中写明「以组件库为准的例外项」，避免研发强行像素级还原不可实现部分 |

## 5. 我的退出标准（对照总计划 §9）

设计不单独「验收 MVP」，但以总计划 MVP 可演示、可提审为前提：  
**关键页有稿或规范可依；主流程状态与边角态有视觉依据；提审截图风格统一。**  
若因设计缺位导致小程序无法按约定时间开工，由我优先补齐 **最小可用设计包**（规范 + 核心页）。
