# P0-2 · 一期派单模式决策

> **对应任务**：产品负责人工作计划 · Phase 0 · P0-2。  
> **目的**：冻结一期 **抢单（公共大厅）**、**玩家点名（客户指定陪玩）**、**管理端指派** 的交付方式，以及 **同一待分配订单** 上 **认领/指派** 的 **互斥与串行裁决**，保证可联调、可追责。  
> **关联文档**：`P0-1-一期业务流程说明（demo）.md`（流程与名词，**非**接口/状态机终稿）；总计划见 `../../三角洲陪玩派单小程序-快速落实计划.md` §3 Phase 2。

---

## 1. 决策结论（一期 / demo）

| 项 | 结论 |
|----|------|
| **公共大厅抢单** | 玩家 **未点名** 时，订单进入 **公共待抢单池**，陪玩在 **大厅** 主动 **抢单/认领**（主路径心智）。 |
| **玩家点名（客户指定陪玩）** | 玩家下单时可 **点名某位陪玩**（客户请求）；该单 **不进入** 对其他陪玩可见的 **公共大厅**（或列表接口不返回给非被点名人）；**仅被点名陪玩** 可 **接单**（与「抢单」共用 **行为 C** 的接单语义，服务端校验身份）。**一期必做**（见 **§1.2**）。 |
| **管理端指派** | 管理员对 **仍待分配陪玩** 的订单 **指定陪玩**（运营/兜底）；**一期必做**（行为 **D**）。 |
| **三者关系** | **不是三选一**：一期 **同时交付** 三种来源；对 **同一订单、同一时刻** 仅允许 **一种** 成功归属路径——见 **§1.1**（公共抢单 vs 管理员指派）、**§1.2**（点名单与公共池隔离）。 |

**一句话**：**未点名** → 走 **公共大厅抢单**；**已点名** → 仅 **该陪玩** 可接；**管理员** 仍可对 **适用订单** 做 **指派**（与公共池抢单 **互斥**，与点名单规则见 **§1.2**）。

### 1.1 公共待抢单池：抢单与管理员指派的互斥（服务端强制）

适用于订单处于 **公共待抢单**（**未**被玩家点名，或产品约定与 **P2-1** 一致的状态名）时，**大厅抢单认领** 与 **管理员指派** 可能 **并发** 触发的情况。

| 规则 | 说明 |
|------|------|
| **串行裁决** | 对**同一订单 ID**，「陪玩抢单认领」与「管理员指派陪玩」须进入**同一互斥域**，**同一时刻仅允许一笔成功提交**。 |
| **成功 / 失败** | 先完成且合法的一方成功，订单进入 **已归属陪玩**；另一方 **必须失败** 且 **明确业务错误码**（见 **`接口表-v0.md`**）。 |
| **推荐错误语义** | 抢单失败：如「订单已被其他陪玩接单」「订单已被管理员指派」；指派失败：如「订单已被陪玩抢单」「订单状态已变更不允许指派」。 |

> **与 §1.2 的关系**：**玩家已点名** 的订单 **不进入** §1.1 的「公共池抢单 vs 管理员指派」并发模型；该单 **仅被点名陪玩** 可接单（§1.2），**管理员是否可覆盖点名** 见 **P0-3 §4.4**（默认一期 **不覆盖**，仅 **取消** 订单）。

### 1.2 玩家点名（客户请求·指定某陪玩）

| 规则 | 说明 |
|------|------|
| **入口** | 玩家 **下单页** 可选 **点名陪玩**（从俱乐部可接单陪玩列表选择或按规则搜索，字段见 **接口表 A**）。 |
| **可见性** | 被点名订单 **不对其他陪玩展示** 为可抢公共单（**B** 列表过滤或独立「我的点名单」由实现选型，产品验收：**非被点名人不可抢**）。 |
| **接单** | **仅被点名陪玩** 可发起 **接单**（**行为 C**，服务端校验 `current_escort == designated`）。 |
| **与管理员指派** | **默认（一期）**：玩家已点名且 **尚未接单** 时，管理员 **不将订单改派给其他陪玩**（避免与客户意愿冲突）；仅可 **取消订单** 等兜底。若俱乐部要求「管理员可覆盖点名」，须 **书面变更** 并改 **P0-3 §4.4** + **P2-1**。 |
| **与公共抢单** | 同一张单 **不会** 同时出现在「公共大厅」与「点名单」；创建订单时即确定 **是否点名**。 |

### 2.2 最小 API 行为清单（一期须实现 · 待映射到接口表）

| 序号 | 行为 | 成功结果（语义） | 失败须区分（示例语义） |
|------|------|------------------|-------------------------|
| A | 玩家创建订单 | 可选带 **点名陪玩 ID**（见 **§1.2**）。**未点名**：进入 **公共待抢单**，可上大厅；**已点名**：进入 **点名单** 逻辑，**不上** 公共大厅 | 参数校验失败、点名对象非可接单陪玩等 |
| B | 陪玩拉取「可接单」列表 | **公共待抢单**（可被该陪玩抢的单）+ 若有 **点名单**，则包含 **点名给当前陪玩** 的单（实现可二合一接口或分栏，见接口表） | 无陪玩权限等 |
| C | 陪玩 **抢单/认领** | **公共单**：同 §1.1 与 D 互斥；**点名单**：仅当被点名陪玩本人，否则拒绝 | 非被点名人抢点名单；已被他人抢单；已被管理员指派；状态非法 |
| D | 管理员 **指派陪玩** | **仅适用于** 产品约定允许指派的 **待分配** 订单（**默认**：**未点名** 的公共待抢单；**已点名** 单见 **P0-3 §4.4**） | 见 §1.1；若违反 §4.4 则拒绝 |
| E | 管理员取消 / 强制状态修正 | 与 P2-1 允许迁移一致 | 权限/状态非法 |

**与 §1.1 / §1.2 的关系**：**公共池** 上 C 与 D 须 **串行裁决**；**点名单** 上仅 **被点名陪玩** 可走 C；D 能否作用于点名单见 **P0-3 §4.4**。

## 3. 能力对比（帮助研发理解）

| 维度 | 公共大厅抢单 | 玩家点名 | 管理端指派 |
|------|----------------|----------|------------|
| 谁发起 | 陪玩 | **客户/玩家**（下单时） | 管理员 |
| 典型场景 | 不指定谁接 | **就要某陪玩** | 无人抢、运营调度 |
| 列表 | 公共 **B** | 仅对被点名人可见（§1.2） | 后台操作 |
| 并发 | 多陪玩抢同一单需防重 | 仅一人可接 | 与 **公共池** 抢单互斥（§1.1） |

## 4. 对各端的含义（实现边界）

### 4.1 小程序 · 玩家

- 下单 → 等接单 → 服务中 → 完成/取消（见 P0-1）。  
- 可 **可选** **点名陪玩**（§1.2）；下单页字段见 **P0-5**、**接口表 A**。  
- 玩家侧展示：**已接单** 前可显示「等待接单」；若已点名可显示「已指定陪玩，等待 TA 接单」类文案（**P1-2**）。

### 4.2 小程序 · 陪玩

- **必须有**：**公共大厅**（可抢列表）+ **点名单**（仅展示给 **被点名陪玩** 的待接单，可与大厅 **同页分栏/筛选**，实现见接口表 **B**）。  
- **抢单/接单**（**C**）：公共单遵守 §1.1；点名单遵守 §1.2。  
- **不要求一期**：复杂排序、多标签筛选（可按 P0-3 删减）。

### 4.3 后端

- **必须有**：**C + D + E**；**§1.1 串行裁决**；状态机合法迁移。  
- **建议有**：操作日志（谁、何时、原/新状态），与总计划 Phase 2 一致。

### 4.4 管理后台

- **必须有**：订单列表、详情、**指派陪玩**（行为 D）、取消、强制状态修正；详情须展示 **是否玩家点名**（字段见接口表）。  
- **指派 UI**：须可选择目标陪玩；**能否对「已点名未接」单改派** 见 **P0-3 §4.4**（默认 **否**）。

---

## 5. 与「仅抢单、无指派/无点名」方案的区别

- 本文 **一期** 要求 **公共抢单 + 玩家点名 + 管理指派** 按 §1 / §1.2 交付；若资源不足须 **书面变更**。  
- **已接单后的改派** 是否一期做：见 **P0-3 §4.3** 与 **P2-1**。
