# P0-3 · 一期范围与不做清单

> **对应任务**：产品负责人工作计划 · Phase 0 · P0-3。  
> **目的**：书面冻结 **一期（demo / MVP 闭环）做哪些、明确不做哪些**，并落锤 **P0-1 §9**、**P0-2 §2.3** 中依赖 P0-3 的未决项，避免范围蔓延与联调口径悬空。  
> **关联文档**：`P0-1-一期业务流程说明（demo）.md`、`P0-2-一期派单模式决策.md`、`../../三角洲陪玩派单小程序-快速落实计划.md`。

---

## 1. 「一期」所指（门禁对齐）

| 术语 | 含义（本文默认） |
|------|------------------|
| **一期** | 自当前立项起，至总计划 **Phase 2 退出**为止：玩家下单（**可点名陪玩**）→ **公共抢单 / 点名单接单 / 管理端指派**（规则见 **P0-2 §1 / §1.2**）→ 状态推进至完成/取消 → 管理后台可见与操作日志；**不含**微信支付、不含完整商业化。 |
| **Phase 3**（上线准备） | 订阅消息、分页体验、提审材料等：见总计划 §3 Phase 3；**不在「一期必做」内**，除非本节 §2 显式勾选。 |

---

## 2. 一期范围（做）

以下 **一期必做**（与 P0-1、P0-2、总计划一致；实现以 **`接口表-v0.md`** + **P2-1 状态机** 为准）。

| # | 能力 | 说明 |
|---|------|------|
| B1 | 微信登录与用户会话 | 玩家/陪玩/管理员（后台）身份可区分；细节 **P1-4 + 接口表**。 |
| B2 | 服务 SKU 展示与下单 | 固定几项即可（游戏/区服/时长/展示价等）；**可选「点名陪玩」**（客户指定），见 **P0-2 §1.2**、**P0-1** §4～§5。 |
| B3 | 抢单大厅 + 陪玩认领 | 陪玩侧主入口见 **P0-2 §1**；并发与错误码见 **P0-2 §1.1、§2.2 行为 C**。 |
| B3a | **管理端指派（待抢单 → 指定陪玩）** | **一期必做**；行为 **D** 见 **`接口表-v0.md`** 与 **P0-2 §2.2**；与抢单 **互斥** 见 **P0-2 §1.1**。 |
| B4 | 订单列表与详情（玩家/陪玩） | 状态展示用词见 P0-1 §2 + Phase 1 文案包。 |
| B5 | 玩家在 **待抢单且未被认领** 时取消订单 | 见 P0-1 §6。 |
| B6 | 陪玩推进状态 | 「开始服务」「完成服务」等，以 **P2-1** 冻结为准。 |
| B7 | 管理后台：订单列表、详情、**待抢单指派**、关键操作日志（至少状态变更可追溯） | 指派为 **一期必做**（与 B3a 一致）；见总计划 Phase 2。 |
| B8 | 管理后台兜底：**取消订单**、**强制状态修正**（在 P2-1 允许迁移内） | 与 B3a 并列于详情/列表操作区；具体接口见接口表。 |
| B9 | 线下结算 | 系统内 **不接微信支付**；见 P0-1 §4。 |

## 3. 不做清单（一期明确不做）

以下 **一期不做**；若要做须 **书面变更单**（产品 + 负责人 + 研发知悉），并同步改接口表与状态机版本。

| # | 不做项 | 备注 |
|---|--------|------|
| N1 | **微信支付 / 平台内收款 / 分账** | Phase 4 再立项；一期禁止在小程序内完成金流闭环。 |
| N2 | **争议仲裁、平台内退款流程、售后工单** | 纠纷线下沟通；系统不提供仲裁状态与退款按钮。 |
| N3 | **评价 / 评分 / 晒单体系** | 可二期；一期订单完结即结束，无评价页。 |
| N4 | **多俱乐部 / 多租户** | 单俱乐部 demo；不做租户隔离与跨店逻辑。 |
| N5 | **陪玩入驻完整审核流**（证照、多级审核） | demo 可 **仅角色标记 + 后台开关**；不做复杂审核工作台，除非书面变更。 |
| N6 | **自动派单算法、智能推荐、派单权重** | 仅大厅列表 + 抢单；排序规则以接口表最简单实现即可。 |
| N7 | **实时 WebSocket 推送**（若总计划列为二期） | 一期可用轮询或手动刷新；与总计划选型一致。 |
| N8 | **发票、对账报表、财务报表** | Phase 4 及以后。 |

### 4.3 管理端「已接单后的改派」（与「待抢单指派」区分）

|----|------|

### 4.4 玩家已点名后，管理员能否「改派」给其他陪玩

## 5. 变更规则

1. **新增「做」**：须补充到 §2，并标注日期、原因、影响接口表/状态机章节。  
2. **从「不做」改为「做」**：须 **书面变更**，并更新 P0-1/P0-2 相关句段（若冲突）。  
3. **§4 拍板变更**：须三方知悉（产品/后端/小程序）并更新 **P0-2** 中引用 P0-3 的句子（若曾写死默认）。
