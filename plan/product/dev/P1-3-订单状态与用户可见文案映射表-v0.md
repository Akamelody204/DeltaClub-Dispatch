## 1. 内部状态枚举（一期）

与 **`../../../docs/STATE_MACHINE.md` §1**、**`../../../docs/SCHEMA.md`** `orders.status`（若有）**一致**；实现可用 `VARCHAR` 存下列 **字符串常量**（大小写敏感，与后端现网一致）。

| 内部值 `status` | 业务含义（对内） |
|-----------------|-------------------|
| `PENDING_GRAB` | 待分配陪玩（公共池 **或** 点名单，见 **§2**） |
| `ACCEPTED` | 已归属陪玩，尚未开始服务 |
| `IN_SERVICE` | 服务进行中 |
| `COMPLETED` | 正常结束 |
| `CANCELLED` | 已取消或强制关闭 |

**说明**：**不在** 状态枚举中区分「公共池 / 点名单」——区分方式为订单字段 **`designatedCompanionUserId`（或等价）是否非空**，见 **§2**。

## 2. 用户可见短文案（真源表）

用于：**订单列表摘要**、**详情页状态行**、**接口 `statusText` 建议返回值**（成功响应里与 `status` 同发）。

**判定列 `点名`**：`designatedCompanionUserId != null`（接口表字段名以 **`接口表-v0.md`** 为准）。

| 内部 `status` | 点名（仅 `PENDING_GRAB` 参与判定） | 用户可见短文案 `statusText` |
|-----------------|-----------------------------------|------------------------------|
| `PENDING_GRAB` | 否（公共待抢单） | `待接单` |
| `PENDING_GRAB` | 是（点名单待接） | `待 TA 接单` |
| `ACCEPTED` | — | `已接单` |
| `IN_SERVICE` | — | `服务中` |
| `COMPLETED` | — | `已完成` |
| `CANCELLED` | — | `已取消` |

### 2.1 多端一致性

| 端 | 规则 |
|----|------|
| **玩家小程序** | 列表/详情按上表展示；详情内 **补充句**（如「已指定陪玩，等待对方接单」）见 **`P1-2-文案包-v0.md` §4.4**，与上表 **不冲突**（长句为说明，短标签仍用上表）。 |
| **陪玩小程序** | **同一订单** 的 **短标签** 与上表 **相同**（含 `PENDING_GRAB` + 点名时的 `待 TA 接单`）。 |
| **管理后台** | 表格「状态」列 **优先** 与上表一致，便于与小程序对照；若需对内补充（如内部备注），用单独列，**不改** 用户可见列措辞。 |

### 2.2 后端对 `statusText` 的填充（建议）

- **`GET /api/v1/orders`**、**`GET /api/v1/orders/:id`**（玩家）：`statusText` = 按 **§2 表** 由 `status` + `designatedCompanionUserId` 计算。  
- **`GET /api/v1/companion/orders/available`**、**陪玩订单列表/详情**：同上，**同一套字符串**。  
- 若某版本后端 **暂不返回 `statusText`**：小程序 **必须** 按 **§2 表** 本地映射（输入：`status` + `designatedCompanionUserId`）。

### 2.3 与大厅卡片角标（「大厅 / 指定」）

**不** 替代 `statusText`。角标为 **陪玩大厅列表** 辅助信息，见 **`P1-2-文案包-v0.md` §4.6**（`大厅` / `指定`）。
