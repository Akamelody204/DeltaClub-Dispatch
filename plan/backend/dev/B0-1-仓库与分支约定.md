# B0-1 · 仓库与分支约定

> **对应任务**：后端负责人工作计划 · Phase 0 · B0-1。  
> **目的**：定 **仓库形态**、**分支策略** 与 **`main` 保护**，便于小程序 / 管理端 / 后端并行开发且主分支始终可部署。  
> **关联产品文档**（已交付）：`../product/dev/P0-1-一期业务流程说明（demo）.md`、`../product/dev/P0-2-一期派单模式决策.md`、`../product/dev/P0-3-一期范围与不做清单.md` —— 一期 **抢单大厅 + 管理端待抢单指派** 均交付（**P0-3 §4.1 B**）；指派与抢单 **互斥** 见 **P0-2 §1.1**；后端 **单仓 server** 内实现订单/锁/指派 API，不为此拆微服务。  
> **依据**：`../三角洲陪玩派单小程序-快速落实计划.md` §3 Phase 0、`../技术栈选型记录.md` §2.2。

## 1. 仓库结构（当前结论）

| 项 | 结论 |
|----|------|
| **形态** | **单仓 monorepo 推荐**：仓库根目录下设 **`server/`** 为 Node 后端（NestJS 或团队统一框架）；与 `miniapp/`、`admin/` 并列，**一期不拆独立后端仓**，避免多仓同步成本。 |
| **若坚持双仓** | 仅当负责人书面要求时，独立 `clubxcx-server` 仓；本文 **分支与 main 保护规则仍适用**。 |
| **server 目录最小内容** | `package.json`、`src/`、`README.md`、`.env.example`、`.gitignore`（含 `.env`）；**禁止**提交含密钥的 `.env`。 |

**与产品的对齐**：业务闭环见 P0-1；**抢单、指派、状态机**均在 `server` 内演进；订单/用户/权限模块为 **一期必建**，不因范围收缩而省略目录（若砍能力须书面变更 P0-3）。

---

## 2. 分支策略

| 分支 | 用途 |
|------|------|
| **`main`** | **可部署**：与总计划「主分支可部署」一致；合并前需通过 CI（若有）与自检。 |
| **`feat/*`** | 功能开发，例：`feat/b0-health`、`feat/b1-wechat-login`。 |
| **`fix/*`** | 线上或 staging 紧急修复（按需）。 |

**约定**：小团队可 **PR 自合并**（与总计划 §7）；**breaking API** 须在合并前在站会或群公告 **显式同步** 小程序/管理端。

---

## 3. `main` 分支保护（执行清单）

| 项 | 要求 |
|----|------|
| 保护分支 | 托管平台（GitHub / Gitee / 自建）对 **`main`** 开启 **禁止强推**、**建议 PR 评审**（至少 1 人，可同角色不同人）。 |
| 合并条件 | **无密钥进库**；`README` 或 `server/README` 能说明如何本地启动（B0-4 完成后补全）。 |
| 标签（可选） | 里程碑打 `v0.1.0-b0` 类 tag，便于回溯。 |

若当前尚无远程仓：**负责人创建远程后** 按上表一次性勾选。
