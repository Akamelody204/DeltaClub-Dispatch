# B0-3 · MySQL 与 Redis 规划

> **对应任务**：后端负责人工作计划 · Phase 0 · B0-3。  
> **目的**：一期 **MySQL 8** 库级约定与 **Redis** 用途清单，与 `技术栈选型记录` 一致，并为 **P0-2 抢单大厅**（Phase 2 并发控制）预留 Redis 能力，避免临时加栈。  
> **关联产品文档**：`../product/dev/P0-2-一期派单模式决策.md`（§1.1 串行裁决、§4.3 后端必须有防重复认领）、`../product/dev/P0-3-一期范围与不做清单.md`（一期不做 WebSocket，Redis 不作实时推送刚需）。  
> **依据**：`../技术栈选型记录.md` §2.3。

## 1. MySQL

| 项 | 约定 |
|----|------|
| **版本** | **MySQL 8**（与选型记录一致）。 |
| **字符集 / 排序规则** | **utf8mb4** + **utf8mb4_unicode_ci**（或团队统一 utf8mb4_0900_ai_ci，全项目一致即可）。 |
| **库名（占位）** | `clubxcx` 或 `clubxcx_dev` / `clubxcx_staging` / `clubxcx_prod`（分环境分库或分 schema 由负责人定，**须在部署文档写死**）。 |
| **引擎** | InnoDB；表需主键；时间字段建议 `DATETIME(3)` 或统一 UTC。 |
| **迁移** | 使用 ORM 迁移（如 TypeORM / Prisma / Knex）或 Flyway 式 SQL；**禁止**仅手工改生产库无记录。 |

**一期核心实体（与总计划 §6 对齐，表名可调整）**：`users`、`orders`、后续 `order_logs` / `operation_logs`（Phase 2）；SKU 可先配置表或枚举。

## 2. Redis 用途清单（一期）

| 用途 | Phase | 说明 |
|------|-------|------|
| **Session / 登录 token 黑名单**（若采用） | Phase 1+ | 与 JWT 方案二选一或组合；见实现设计。 |
| **派单互斥 / 抢单防重** | Phase 2 | 对齐 **P0-2 §1.1**：同一 `orderId` 抢单与（若未来开放）指派进入 **同一互斥域**；可用 Redis 分布式锁 **或** DB 行锁/乐观锁，**实现前在 B2-2 记录最终选型**。 |
| **限流** | Phase 3 | 按 IP 或用户 ID 滑窗/令牌桶（总计划 Phase 3）。 |
| **简单队列**（可选） | 后续 | 一期不强制；异步任务可先用 DB 轮询或延后。 |

**非目标**：一期 **不把 Redis 当主存储**；订单真相仍在 MySQL。

---

## 3. 与 P0-3「不做」的边界

| P0-3 项 | 后端含义 |
|---------|----------|
| N7 无 WebSocket 刚需 | 不为此单独上 Redis Pub/Sub；列表刷新以前端轮询或手动刷新为主。 |
| 无微信支付 | 无支付回调幂等键存 Redis 的硬性要求（Phase 4 再评估）。 |

| 环境 | 建议 |
