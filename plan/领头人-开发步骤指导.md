# 领头人 · 开发步骤指导（执行顺序与门禁）

> **文档角色**：本项目由领头人**对交付与事故负总责**；本指导规定 **先谁、后谁、哪几步可并行、何时才能进入下一阶段**，并与各负责人工作计划中的**任务编号**对齐，便于拆 issue 与追责。  
> **真源依据**：`三角洲陪玩派单小程序-快速落实计划.md`、`技术栈选型记录.md`，以及 `product/`（**执行产出见 `product/dev/`**）、`backend/`、`design/`、`miniapp/` 下各负责人计划。

---

## 1. 领头人职责（出事我负责时的「我」具体做什么）

| 事项 | 动作 |
|------|------|
| **门禁** | 每 Phase 结束前对照本文「退出门禁」勾选；未达标**禁止**全员宣布进入下一阶段。 |
| **阻塞升级** | 产品未定派单模式/范围、域名未就绪、设计阻塞小程序开工等，**当日**牵头拉 15～30 分钟对齐会或书面拍板。 |
| **接口与状态机变更** | 任何 breaking 变更必须在**当日站会**宣布，并督促更新接口表版本号。 |
| **风险登记** | 俱乐部未回复、微信类目不明、无管理端人力等，记入风险表并同步负责人。 |
| **发版** | 预演通过 + 产品审核材料齐 + 后端生产 checklist 勾完前，**不批准**提交微信审核或上生产。 |

各角色**仍按各自计划承担专业责任**；领头人负责**顺序、取舍、升级与最终放行**。

---

## 2. 全局执行原则（全员遵守）

1. **产品真源**：范围、状态机、接口表、审核文案以产品维护的文档为准；变更必须可追溯。  
2. **先冻结再深写逻辑**：**P0-2（派单模式）+ P0-3（一期范围/不做清单）** 书面落地前，后端 **B2-2/B2-3**、小程序 **M2-1/M2-2** 不得按「最终版」写死（可先做登录与订单草稿）。  
3. **主分支可联调**：领头人每日关注——**staging API 可访问 + 小程序工程可指向 staging**；长期 mock-only 视为红色风险。  
4. **管理端**：若无人专职前端，由领头人指定 **后端兼 Vue 极简后台** 或外协，并写入排期；**不静默假设「后台会自动出现」**。

## 3. Phase 0：对齐与准备（约 3～5 天）

### 3.1 第 1～2 天（必须先串起来）

| 顺序 | 谁 | 执行计划中的步骤 | 说明 |
|:----:|----|------------------|------|
| ① | **产品** | **P0-1** → **P0-2** → **P0-3** | 俱乐部口径与**派单模式二选一**、**不做清单**书面化；**阻塞全局**，领头人每日跟进直到闭合。 |
| ② | **产品** | **P0-4** | 审核自检表 v0；与 **P0-1** 可部分并行，但须在 Phase 0 内完成。 |
| ③ | **设计** | **D0-1**（依赖产品主路径） | 与产品 **P0-5** 对齐关键页清单；产品 **P0-1** 有初稿后即可启动，不必等 Phase 0 最后一天。 |
| ④ | **后端** | **B0-1 → B0-4** | 仓库、环境规划、健康检查可跑；**可与 ① 并行**。 |
| ⑤ | **小程序** | **M0-1 → M0-2 → M0-3** | AppID、空壳工程、环境切换说明；**可与 ②③④ 并行**。 |
| ⑥ | **设计** | **D0-2** | 线框/信息结构；依赖 **D0-1** 闭合。 |
| ⑦ | **全员** | 选型与接口文档位置对齐会 | 对应总计划 Phase 0「全员」行 + 后端 **B0-5**、小程序 **M0-4**；领头人**必须到场拍板**文档 URL 与仓库策略。 |

## 4. Phase 1：骨架（约 1～2 周）

### 4.1 推荐顺序（周维度）

**Week A（优先打通链路与契约）**

| 顺序 | 谁 | 步骤 | 说明 |
|:----:|----|------|------|
| ① | **产品** | **P1-1 接口表 v0**（启动） | 领头人督促：**先出「登录 + 用户 + 订单 CRUD 草稿」** 最小表，再迭代字段；避免后端空等。 |
| ② | **后端** | **B1-1 → B1-2** | 微信登录与鉴权；可与 ① **半天内交错**：产品给首版字段，后端先落用户表。 |
| ③ | **后端** | **B1-3** | 订单表 v0 + CRUD；对齐接口表 v0。 |
| ④ | **小程序** | **M1-1** | 登录链路；依赖 **B1-1** 可用（可用 dev 环境先联调）。 |
| ⑤ | **小程序** | **M1-2 → M1-3 → M1-4** | 拦截、玩家骨架、陪玩骨架；依赖产品 **P1-4** 分流规则。 |
| ⑥ | **产品** | **P1-2、P1-3、P1-4** | 与 **P1-1** 并行推进，**P1-4** 须在 **M1-4** 开发中段前闭合。 |

**与设计并行**

| 谁 | 步骤 | 说明 |
|----|------|------|
| **设计** | **D1-1**（规范 + 核心页 或 4～6 页高保真） | Phase 1 **第 3～5 个工作日内**应达到「小程序可照着做」的**最小包**（总计划与设计计划中的「方案 A/B」）。 |
| **设计** | **D1-2、D1-3** | 与 **D1-1**、产品 **P1-2** 穿插完成。 |

**联调关门**

| 顺序 | 谁 | 步骤 | 说明 |
|:----:|----|------|------|
| ⑦ | **后端** | **B1-5** | staging 部署 HTTPS。 |
| ⑧ | **小程序** | **M1-5** | 微信后台合法域名 + 真机连 staging。 |
| ⑨ | **后端** | **B1-4** | 管理端鉴权占位（若本期有管理端演示）。 |

### 4.2 Phase 1 退出门禁

- [ ] **B1-1～B1-3、B1-5** 完成；真机/开发者工具 **登录成功**  
- [ ] **M1-1～M1-5** 完成；玩家/陪玩骨架与 **P1-4** 一致  
- [ ] **P1-1** 与实现字段对齐（无「仅口头约定」）  
- [ ] **D1-1** 最小设计包已交付，或领头人已书面同意「先规范后补高保真」的截止日期

## 5. Phase 2：派单闭环 MVP（约 2～3 周）

### 5.1 进入本 Phase 的硬前置

- **产品 P2-1**：订单状态机**冻结**（图 + 规则）；**领头人确认签字或文档版本号**。  
- **P0-2 派单模式**已写入接口表影响范围（抢单大厅 vs 指派）。

### 5.2 执行顺序（可部分并行）

| 轨道 | 谁 | 步骤 | 说明 |
|------|----|------|------|
| **A 后端核心** | 后端 | **B2-1**（状态机）→ **B2-2**（派单策略 + 锁）→ **B2-3** | **B2-1 先于** 小程序最终对接逻辑；**B2-2** 依赖 P2-1。 |
| **B 后端管理** | 后端 | **B2-4**（操作日志）可与 **B2-3** 并行；**B2-5** 与管理端联调时闭合。 |
| **C 小程序** | 小程序 | **M2-1**（玩家）与 **M2-2**（陪玩）在 **B2-3 接口可用**后优先对接；**M2-3** 全程穿插。 |
| **D 管理端** | 后端兼或专人 | 管理端 Web：对齐总计划 Phase 2「管理后台」行；API 以 **B2-3/B2-5** 为准。 |
| **E 产品验收** | 产品 | **P2-2** 每 2～3 天走查 + 已知问题列表 | 领头人盯 **P0 级当日认领**。 |
| **F 设计** | 设计 | **D2-1、D2-2** | 边角态与状态变体；**D2-3** 走查可选参与。 |
| **G 产品** | 产品 | **P2-3、P2-4** | 后台验收与边角对齐。 |

### 5.3 Phase 2 退出门禁

- [ ] 玩家下单 → 陪玩接单（或管理指派）→ 服务中 → 完成/取消 **全链路真实数据**  
- [ ] 管理端可查单、最小干预（指派/改派/状态修正）**或** 领头人已批准「俱乐部暂用 SQL/接口工具」的临时方案并记风险  
- [ ] **无双接单**（后端 B2-2 验证说明可查）  
- [ ] 产品 **P2-2** 中无未分配责任的 **P0**

## 6. Phase 3：可用性与上线准备（约 1～2 周）

| 顺序 | 谁 | 步骤 | 说明 |
|:----:|----|------|------|
| ① | **后端** | **B3-1 → B3-2 → B3-3** | 分页、限流、生产配置与备份；**B3-3** 须在预演前完成。 |
| ② | **小程序** | **M3-1** | 订阅消息；与产品 **P3-4**、后端 **B3-4** 对齐节点与幂等。 |
| ③ | **小程序** | **M3-2 → M3-3** | 分页刷新、审核配合。 |
| ④ | **产品** | **P3-1 → P3-2 → P3-3** | 自检表、提审包、预演脚本；**P3-3** 领头人指定预演日期。 |
| ⑤ | **设计** | **D3-1、D3-2** | 截图与走查。 |
| ⑥ | **全员** | 预演 → 问题闭环 | 后端 **B3-5**、小程序 **M3-4**；**领头人批准**后才可提交审核/上生产。 |

### Phase 3 退出门禁

- [ ] 产品 **P3-1～P3-3** 完成；提审材料可提交  
- [ ] 后端生产 checklist 与备份策略已确认  
- [ ] 预演问题 **P0 清零**（或领头人书面接受残留风险）

| 频率 | 内容 |
