# 三角洲陪玩派单小程序 · 技术栈选型记录

> **文档性质**：在他人推荐基础上的**批判性采纳** + **本项目倾向结论**，便于立项与后续变更可追溯。  
> **优先级**：开发效率、团队可维护性、微信审核与上线路径；不追求「全家桶」。

### 项目范围（用户端 · 已确认）

| 项 | 当前决策 |
|----|----------|
| **用户端** | **仅微信小程序**（不做支付宝小程序、不做对外 H5 业务站、不做 App 用户端，除非后续单独立项）。 |
| **管理端** | **PC Web 管理后台**（浏览器）照常开发，不属于「多端小程序」讨论范围。 |
| **官网/落地页** | 如需品牌展示页，独立按需；与小程序技术栈无绑定。 |

---

## 1. 对「他人推荐」的批判性看法

### 1.1 前端：UniApp + Vue3 + TypeScript

| 说法 | 更稳妥的理解 |
|------|----------------|
| 「一次开发多端，效率提升 50%」 | **方向对，比例不可信**。多端仍有差异（各小程序 API、支付、审核规则），需 `条件编译` 与真机联调，实际节省的是「若真要做 App/H5 第二端」的重复开发；**只做微信小程序时，相对原生的优势没有数字上那么夸张**。 |
| Vue3 + TS | **可行**。注意：uni-app 下 TS 类型与部分第三方库仍需磨合；小团队若 TS 不熟，可 **Vue3 + JS 先跑通**，再渐进加类型。 |
| uView Plus / ColorUI | **合理**。选型时看：**Vue3 支持、更新频率、issue 活跃度**；避免选已停更库。 |

**结论**：若**确定近期要微信 + 支付宝（或 H5）**，UniApp 是高效选项；若**仅微信、且团队更熟原生**，原生 + 自建后端同样能快，不必为「多端」强行 UniApp。

---

### 1.2 备选：微信原生 + 云开发

| 优点 | 风险 / 局限 |
|------|----------------|
| 免运维、与微信一体、原型极快 | **复杂派单、状态机、管理后台、分账** 时，云函数与权限模型会变复杂；**跨平台迁移成本高**；部分团队觉得调试与版本管理不如传统后端顺手。 |

**结论**：适合 **MVP 验证 / 极小规模**；若已规划 **独立管理端、清晰后端权限与审计**，更建议 **自建 API 服务**（Node/Java/PHP 任一），云开发可作**部分能力补充**而非全盘依赖（除非团队明确押注全栈云开发）。

---

### 1.3 后端：Node+Mongo vs ThinkPHP+MySQL+Redis vs 云开发

| 方案 | 批判性评价 |
|------|------------|
| **Node + Express/Koa + MongoDB** | 「灵活」对**聊天记录、日志**友好，但**订单、支付、派单状态机**强一致场景下，**关系型更省心**；Mongo 不是「必然更快」，缺经验时易在事务与建模上踩坑。 |
| **ThinkPHP 6/7 + MySQL + Redis** | **国内外包与二开市场常见**，文档与案例多；**复杂订单 + 后台** 很合适。若团队无人熟悉 PHP，不必为「生态」硬选。 |
| **微信云开发** | **Vibe 运维**最好；业务复杂后常面临 **复杂查询、跨端、精细权限** 的边界。 |

**结论**：**不要**「为了灵活」在核心业务上用 Mongo **替代** MySQL；若确需文档型数据（如大量非结构化消息体），可 **MySQL 主业务 + Mongo 辅助** 或 **先全放 MySQL/JSON 字段，量大了再拆**。

---

### 1.4 数据库组合：MySQL + MongoDB + Redis 全上

| 问题 | 建议 |
|------|------|
| 三库并行 | **中小团队一期维护成本高**，联调、备份、监控都要三套心智。 |
| 聊天记录 | 早期 **MySQL 分表 / 腾讯云 IM 托管** 往往比自建 Mongo 更省事；**日活与消息量上来**再评估 Mongo 或专用 IM 存储。 |

**结论**：**一期推荐 MySQL + Redis**；Mongo / 专用消息存储 **按需二期引入**，写在路线图里即可。

---

### 1.5 第三方服务清单

| 项 | 评价 |
|----|------|
| **腾讯云 IM / 网易云信** | 能力正确；**成本 + 审核 + 内容合规**都要考虑。**MVP** 可用 **站内消息表 + 订阅消息 + 企业微信/群运营** 替代一部分；**强社交、强实时**再上 SDK。 |
| **微信支付** | **商业化必选项**；分账需主体与资质，与产品节奏对齐。 |
| **腾讯位置 LBS** | **三角洲陪玩多为线上匹配**，「附近陪玩」未必是核心；**若产品明确要做地理派单**再加，避免无效集成。 |
| **WebRTC / 语音 SDK** | **高级功能**；合规与成本都高，**明确不做语音开黑一期可完全不选**。 |
| **Redis** | **同意**，会话、热点、派单锁、限流都好用。 |

## 2. 本项目倾向结论（可视为「当前敲定版」）

> 以下按 **用户端已确认：仅微信小程序**（见文首「项目范围」）且 **开发效率优先** 假设。若未来要极简运维且业务极简单，可再评估「云开发为主」并在变更记录中说明。

### 2.1 前端（小程序）

| 层级 | 选型 | 说明 |
|------|------|------|
| 框架 | **UniApp + Vue3** | 与 Vue 管理端技术统一、生态成熟；**TypeScript 建议为「推荐但非首日强制」**。 |
| 目标端 | **仅发布微信小程序** | 不为支付宝/H5/App 单独排期；若日后商务要求扩展，再评估 **uni-app 条件编译** 或 **另起端** 的成本。 |
| UI | **uView Plus**（优先）或 **官方 uni-ui** | 以 **Vue3 + 持续维护** 为准则；ColorUI 需确认与当前 uni-app 版本匹配再选。 |

**与微信原生对比**：在「只做微信」下，**原生小程序**同样合理（调试路径更直接）。当前仍倾向 **UniApp** 的主因是 **与管理后台同属 Vue 生态、组件与习惯可复用**；若团队全员更熟原生，可改选原生，**不必为「未来多端」强行 UniApp**（未来若真要多端再议）。

---

### 2.2 后端（API 服务）

| 层级 | 选型 | 说明 |
|------|------|------|
| 运行时 | **Node.js** | 与前端语言接近，招人/协作友好；框架建议 **NestJS**（结构清晰）或 **Koa**（更轻），Express 亦可。 |
| 备选（团队若更熟 Java） | **Spring Boot + MySQL + Redis** | 与 Node 二选一即可，**不混用双后端**。 |
| 不推荐一期默认 | **ThinkPHP** | 仅当团队已有 PHP 主力时采纳；不为「文档多」单独换栈。 |
| 云开发 | **不作为一期核心后端** | 除非团队明确全栈云开发且接受锁定；可作 **静态资源、触发器类** 补充（可选）。 |

---

### 2.3 数据与缓存

| 组件 | 选型 | 说明 |
|------|------|------|
| 主库 | **MySQL 8** | 用户、订单、派单、配置、审计日志结构化存储。 |
| 缓存 | **Redis** | Session、热点、派单分布式锁、简单队列、限流。 |
| 文档库 | **一期不上 MongoDB** | 聊天记录先 **MySQL 或 IM 厂商托管**；需要时再引入。 |

---

### 2.4 管理后台

| 选型 | 说明 |
|------|------|
| **Vue3 + Element Plus** | 与 UniApp（Vue）技术栈统一，组件心智一致。 |

---

### 2.5 第三方服务（分期）

| 阶段 | 服务 |
|------|------|
| **一期（MVP）** | 微信登录、（可选）订阅消息；支付可 **线下记账** 或 **微信支付** 二选一由商务决定。 |
| **二期** | 微信支付分账、腾讯云 IM 或网易云信（若产品强依赖私聊）、风控与报表。 |
| **按产品再定** | 腾讯位置服务（仅当「附近陪玩」为刚需）、WebRTC/语音 SDK（明确高级场景再做）。 |

---

### 2.6 基础设施

| 项 | 说明 |
|----|------|
| 部署 | 云主机 / 容器均可；**HTTPS + 合法域名** 配齐小程序后台。 |
| 文件 | **腾讯云 COS**（或同云厂商 OSS）存头像、凭证图。 |
| 实时 | 一期 **订阅消息 + 轮询**；派单强实时再上 **WebSocket** 或 IM 自带通道。 |

## 3. 与「他人推荐」对照摘要

| 维度 | 他人推荐 | 本记录倾向 |
|------|----------|------------|
| 小程序 | UniApp + Vue3 + TS | **同上，TS 渐进** |
| UI | uView Plus / ColorUI | **uView Plus 优先**，ColorUI 需验证版本 |
| 后端轻量 | Node + Mongo | **Node + MySQL**，Mongo 延后 |
| 后端企业 | ThinkPHP + MySQL + Redis | **认同 MySQL+Redis**；PHP **按团队栈选** |
| 云开发 | 强推荐 | **可作补充，不作一期核心**（除非团队押注） |
| 三数据库 | MySQL+Mongo+Redis | **一期 MySQL+Redis** |
| IM / LBS / 语音 | 列得全 | **分期上，避免 MVP 全家桶** |
