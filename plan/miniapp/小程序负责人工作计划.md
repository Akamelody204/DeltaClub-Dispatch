# 小程序负责人工作计划（对总计划负责）

> **角色**：负责 [`三角洲陪玩派单小程序-快速落实计划.md`](../三角洲陪玩派单小程序-快速落实计划.md) 中 **「小程序」** 所列全部事项（与总计划 §2「小程序」一致：玩家端 + 陪玩端页面，或同端角色切换等）。  
> **问责**：上述小程序范围内若出现**未交付、主流程联调阻塞、体验事故（重复提交、无鉴权拦截、错误态缺失）、微信侧配置遗漏（合法域名、类目、提审材料配合）**，由本角色牵头定位、修复与复盘；**业务规则与审核文案真源**仍以产品与总计划为准。  
> **依据文档**（真源）：`plan/三角洲陪玩派单小程序-快速落实计划.md`、`plan/技术栈选型记录.md`。

**总计划里小程序要对齐的章节（避免歧义）**：§2 角色表「小程序」；§3 各 Phase 表格中 **「小程序」** 列全部任务；§4 小程序技术选型（原生 / uni-app 等与团队最终选定一致）；§6 中与前端消费相关的接口约定；§7 站会中接口 breaking、环境联调配合；§8 审核、并发体验（前端防重复点击等）；§9 验收标准中登录、下单、陪玩接单等**用户端可感知**部分。

**范围边界**：**PC 管理后台 Web** 不在本角色主责内（总计划已单列）；若人力紧张由他人兼做，本角色仍须配合接口联调与验收中的「小程序侧」条目。

## 1. 总原则（我承诺遵守）

1. **主分支可编译可连真后端**：每天至少保证小程序工程能指向 staging（或约定 dev）并成功发起带 token 的请求，避免长期纯 mock 与主计划脱节。  
2. **鉴权与路由一致**：无有效 token 按产品规则拦截或跳转登录；玩家/陪玩可见页面与接口权限与产品文档一致，不依赖「用户自觉不调接口」。  
3. **请求与配置不写死密钥**：仅使用后端下发的会话方案；`AppSecret` 等永不在小程序代码或仓库中出现。  
4. **微信规范与体验红线**：合法域名、业务域名（若用到 web-view）、订阅消息类目与模板，按微信文档与产品审核自检表执行；主流程 loading、错误 toast、防重复提交必须到位。  
5. **技术栈按选型记录与负责人最终拍板执行**：一期仅微信小程序；在 **原生** 与 **uni-app (Vue3)** 等选项中，团队选定后本角色统一工程规范（目录、请求封装、环境切换），不混用两套脚手架。

| 序号 | 任务 | 产出 | 完成标志 |
|------|------|------|----------|
| M0-1 | 注册/绑定小程序 **AppID**；开发者工具与仓库权限就绪 | 团队可见说明（不含密钥） | 本机可打开工程 |

### Phase 1：骨架（约 1～2 周）

| 序号 | 任务 | 产出 | 完成标志 |
|------|------|------|----------|
| M1-1 | **登录链路**：启动/登录页拉 `code`、调后端换 token、**storage** 持久化；请求封装自动带 token | 登录相关页面 + `request` 封装 | 真机或开发者工具登录成功 |
| M1-2 | **无 token 拦截**：按产品规则跳转登录或提示 | 全局逻辑或路由守卫 | 未登录不可误进受保护页 |
| M1-3 | **玩家端骨架**：首页、下单（可先静态或半静态表单）、订单列表、我的；路由可切换 | 页面骨架 | 请求带 token，接口 401 有统一处理 |
| M1-4 | **陪玩端骨架**：待接单列表（可先 mock）、进行中订单入口；与 **Tab/入口分流** 规则一致 | 页面与路由 | 与产品 P1-4 规则一致 |
| M1-5 | 配合后端完成 **小程序后台 request 合法域名**（staging HTTPS）配置与联调验证 | 配置截图或 checklist 勾选 | 真机访问 staging API 成功 |

**Phase 1 我的退出标准**：总计划 Phase 1 中「小程序」行全部满足；与 §9 中「真实微信环境可登录」对齐。

## 3. 核心页面与接口消费（小程序 checklist，对齐总计划 §6）

**页面/能力最小集（用户端）**

- 登录 / 会话恢复（与后端登录、刷新策略一致）  
- 创建订单、订单列表、订单详情  
- 陪玩：可抢列表或待接列表、接单、状态相关操作（与接口表一致）  
- 我的：基础资料、角色入口（若同端多角色）  

**工程与质量自检**

- 所有受保护接口经统一封装带 token  
- 关键操作防抖/节流或提交中禁用  
- 订单状态展示使用产品「用户可见名称」映射

## 5. 依赖与升级（提前喊阻塞）

| 依赖 | 若缺失的影响 | 我的动作 |

## 6. 风险登记表（小程序视角，与总计划 §8 呼应）

| 风险 | 小程序侧应对 |
|------|----------------|
| 微信审核不通过 | 配合产品自检表；避免违规导流、私下交易话术；类目与功能一致 |
| 重复下单/重复接单（体验层） | 按钮提交态、请求幂等配合（以后端为准）、结果返回前禁用 |
| 订阅消息用户拒授权 | 产品已定降级文案；不阻断主流程 |
| 环境混用导致误连生产 | 构建配置与图标/名称区分；发版 checklist |
